# תכנון v2 — משפכים לפי תוכנית + מבט רוחבי (להחליף את האקסל)

מבוסס על: ההודעה הקולית של טל (PTT-20260621, 8:50 דק', תומללה), הערות טל בכתב, ותוספות נועם (21/06/2026).

## מה טל אמרה — התובנה המרכזית

האפליקציה הנוכחית בנויה על **משפך אחד גלובלי**. אבל לטל יש **שלוש תוכניות עם שלושה משפכים שונים לגמרי**, ו"הרשמה" משמעותה דבר אחר בכל תוכנית. בנוסף, ג'אפא (ה-CRM) נותנת לה מבט **על-בן-אדם** בלבד — בשביל המבט **הרוחבי** (מי עוד צריך טיפול, מה מצב כולם) היא חוזרת לאקסל. **האפליקציה תחליף את האקסל רק אם תיתן גם מבט רוחבי ברור וגם מבט נקודתי ברור — ויזואלית ופרקטית.**

## 2. המשפכים — שלב-אחר-שלב (מהתמלול)

### הדיאלוגי (המלא)
1. **פנייה חדשה** (אידיאלית מיובאת אוטומטית מה-CRM/מייל; ידני/וואטסאפ גם בסדר)
2. **יצירת קשר / נשלח חומר** (לנסח מחדש: "נוצר קשר ונשלח חומר"; תבנית ההודעה להעתקה ✓)
3. **סטטוס / שיחה** — מתלבט? לא עומד בקריטריונים? → לשנות סטטוס או **למחוק פנייה**
4. **מסמכים + דמי הרשמה (300)** — צ'ק-ליסט מסמכים (ראו §3). **שער: בלי 300 אי-אפשר לקבוע ריאיון**
5. **ריאיון** — נקבע כשהמסמכים מלאים
6. **ועדת קבלה + חוות דעת מהמראיין** → התקבל / לא התקבל → **מכתב קבלה**
7. **מקדמה** → = **נרשם** (תפס מקום). כאן ההרשמה הושלמה.
8. ✗ לא מתאים / נדחה / לא התקבל (סגור)

### תעודה (כוח הצמצום) — פחות שלבים
1. **פנייה**
2. **שיחה ראשונית / יצירת קשר** — איך הגיב? מתעניין? שאל שאלות? לא מתאים?
3. **ריאיון עם הרב רונן + חוות דעת**
4. **התקבל?**
5. **דמי הרשמה / מקדמה** — *אחרי* הריאיון (לא תנאי-סף לריאיון, כי זה חלק משכר הלימוד)
6. **יתרת שכר הלימוד** → נרשם

### אח"ד — כמעט בלי שלבים
1. **פנייה**
2. **תשלום מלא** → = נרשם

## 5. מה צריך להשתנות באפליקציה

### Phase A — מודל הליבה: משפך לכל תוכנית *(העיקרי, הכי בעל-ערך)*
- `config.gs`: אובייקט `PROGRAMS` — לכל תוכנית: שלבים משלה, עוגן-הרשמה, מראיין, צבע, שערים.
- **נרמול שמות תוכנית** ב-importer: שמות הלשוניות הארוכים → 3 תוויות נקיות (`הדיאלוגי`/`תעודה`/`אח"ד`) + `מחזור` נפרד. (לדוגמה "נרשמים תעודה מחזור כב תשפז" → תוכנית=תעודה, מחזור=כב תשפז.)
- `computeRow_` הופך ל**תלוי-תוכנית**: השלב והפעולה-הבאה נגזרים מהמשפך של אותה תוכנית.
- עמודות ה-Kanban משתנות לפי המשפך של התוכנית שסוננה.
- שדה "תוכנית" בכרטיס = רשימה נפתחת של **שלוש התוכניות הפעילות** בלבד (משדרג את הערה #1 שכבר מומשה).

### Phase B — מבט נקודתי (הכרטיס)
- **צ'ק-ליסט מסמכים** (§3): תיבות ✓ נפרדות, לא-מעכבות; דמי הרשמה כרובריקה.
- **שער דמי-הרשמה→ריאיון** בהדיאלוגי (אזהרה, §4).
- **כפתור מחיקת פנייה** (apiDelete חדש).
- **דריסת שלב ידנית** — השלב מחושב אוטומטית כברירת מחדל, אבל טל יכולה לכפות שלב (גמישות §4).
- **תזכורת "לוודא שהכסף הגיע"** במקום checkbox "נרשם ב-CRM" (היא אמרה שזה פחות נחוץ).
- ניסוח "נשלח חומר" → "נוצר קשר ונשלח חומר".

### Phase C — מבט רוחבי (להחליף את האקסל) *(לב הבקשה של נועם)*
- **טבלת-על לכל תוכנית**: שורה = אדם, עמודות = הממדים שטל עוקבת אחריהם רוחבית:
  שם · שלב · פעולה הבאה · קשר נוצר · מסמכים (x/y ✓) · דמי הרשמה ✓ · ריאיון · התקבל · מקדמה ✓ · ימים מאז עדכון · תזכורת.
- **צבע לכל תא**: ירוק=בוצע · כתום=ממתין/נדרש · אדום=תקוע/באיחור.
- מיון לפי שלב או לפי ימים-מאז-עדכון; הפעולה-הנדרשת מודגשת.
- זה המסך שמחליף את "החזרה לאקסל": סריקה מהירה של "מי עוד צריך טיפול ומה מצבו".
- ה-Kanban נשאר כמבט-זרימה ויזואלי; הטבלה = המבט הרוחבי המפורט; הכרטיס = המבט הנקודתי.

### Phase D — ויזואל ועיצוב
- ליטוש עיצוב הדף לבהירות בשני המבטים; שמירה על הפלטה הרגועה + צבע לכל תוכנית.

### Phase E — *(אופציונלי, בקשת נועם)* קליטת ייצוא ה-Excel מג'אפא
ג'אפא יודעת לייצא ל-Excel. שתי דרכי אוטומציה:
1. **תיקיית Drive משותפת + טריגר** — טל מפילה את הקובץ, טריגר זמני ממיר ל-Sheet וקולט. אוטומטי מלא.
2. **העלאה ידנית באפליקציה** — קלט קובץ ב-Web App → פענוח → קליטה. נפילה-לאחור.
- שתיהן ישיגות ב-Apps Script (המרה דרך Drive API). **תלוי בשאלה כמה זה באמת חוסך** מול ההזנה הידנית — אם לא מסייע, מוותרים (כדברי נועם).
- ⚠️ פתוח: צריך דגימת קובץ ייצוא אמיתי מג'אפא כדי לדעת את הסכמה.
