# מעקב פניות והרשמה — תוכניות הדגל (בית הספר הדיאלוגי)

תת-פרויקט בתוך `secretary-project/`. מערכת web למעקב אחר פניות להרשמה לשתי תוכניות הדגל, מבוססת Google Sheets כ-backend, בקו האדריכלות הקיים (Apps Script + Sheets, חינם, בלי שרת).

> **סטטוס:** Phase 1 — **פרוס ועובד** תחת חשבון `merkazrotenbergkh@gmail.com` (2026-06-17). 62 רשומות מיובאות נקי. הקוד ב-`apps-script/`, הוראות ב-[INSTALL.md](INSTALL.md). הקובץ החי **לא נגענו בו**. ראו "עדכון מצב" בתחתית.

### קבצים (apps-script/)
| קובץ | תפקיד |
|---|---|
| `config.gs` | מזהים, פרמטרים, סכמת `פניות`, מילון נרדפות-כותרות, שלבי משפך — מקור-אמת יחיד |
| `lib.gs` | גישה ל-App DB, נרמול, חישוב שלב/פעולה/תזכורת, מפתח זהות ל-upsert |
| `setup.gs` | הקמה חד-פעמית של ה-App DB (לשוניות, ולידציה, עיצוב מותנה, תפריט) |
| `importer.gs` | סנכרון חד-כיווני קובץ-חי→App DB; מונה לשוניות (כל אחת=תוכנית), זיהוי כותרות גמיש |
| `Code.gs` | שרת ה-Web App (board / update / create / dashboard / import / send-material) |
| `index.html` | ממשק RTL: קנבן לפי שלב + דשבורד + כרטיס עריכה; פלטה רגועה (slate/teal) |

**הרצה:** ראו [INSTALL.md](INSTALL.md) — מדביקים תחת חשבון רוטנברג, `setup()` → `runImport()` → Deploy as Web App. ~15 דק'.

## 1. מטרה

לעקוב ולעדכן את משפך הפניות לשתי תוכניות הדגל — מרגע הפנייה ועד הרשמה משלמת — דרך ממשק נוח, עם הזנת נתונים חצי-אוטומטית (המזכירה מסמנת עובדות אטומיות; השלב/הפעולה-הבאה/התזכורות מחושבים לבד).

**התוכניות:** **כל לשונית בקובץ החי = תוכנית בפני עצמה** (נועם, 2026-06-17). אין מספר קבוע — ה-importer יעבור על כל הלשוניות דינמית, וכל לשונית תהפוך לערך ב-`תוכנית`. בין התוכניות שזוהו: *כוח הצמצום* (תעודה חד-שנתית, ראיון עם הרב ד"ר רונן בן-דוד), *תעודה תשפ"ז*, *מחזור כ"א תשפ"ו*, ועוד.

**משתמשים:** טל (מזכירה) · מיכל (מנהלת המכון) · אנשי צוות ניהול כולל נועם. **הפונים לא מקבלים גישה לקובץ/אפליקציה.**

**חשבון מריץ:** חשבון ה-Gmail של **מכון רוטנברג** (החשבון שממנו מקורו של הקובץ) — תחתיו ייכתב ויורץ ה-Apps Script וממנו יישלחו ההודעות. **לא** החשבון של נועם.

## 2. החלטות יסוד (נועם, 2026-06-17)

1. **היעד הסופי = ממשק web**, ובסוף עובדים **רק דרכו**.
2. **אסור לפגוע בקובץ החי** — הוא מקור-האמת היום. כל בנייה היא **מקבילה**, עד שהמערכת מלאה.
3. כי הבנייה מקבילה — **חובה מסלול מיגרציה/איחוד מפורש** ל-cutover (מעבר לעבודה דרך האפליקציה בלבד).
4. כרגע **אין גישה ל-CRM** של המכון — רק לגיליון. הזרם חד-כיווני; הגיליון/האפליקציה הם מקור-האמת למשפך, ה-CRM מחזיק את רשומת הלקוח הרשמית.
5. **כל לשונית בקובץ = תוכנית.** ה-importer מונה לשוניות דינמית; כל לשונית → ערך ב-`תוכנית`. מתרחב ל-N תוכניות בלי שינוי קוד.
6. **משתמשים:** טל (מזכירה), מיכל (מנהלת), צוות ניהול כולל נועם. הפונים — ללא גישה.
7. **חשבון מריץ = מכון רוטנברג** (מקור הקובץ). ה-Apps Script ייכתב/יורץ תחתיו; ההודעות יישלחו ממנו.

## 4. ארכיטקטורה — שלושה רכיבים

```
┌─────────────────────┐     importer (חד-כיווני,        ┌──────────────────────┐
│  הקובץ החי           │ ──  לקריאה בלבד, לפי דרישה)  ──▶ │  App DB (גיליון חדש)  │
│  (לא נוגעים בו)       │                                  │  טבלת "פניות" מנורמלת │
│  המזכירה ממלאת       │                                  └──────────┬───────────┘
└─────────────────────┘                                              │ read/write
                                                          ┌──────────▼───────────┐
                                                          │  Web App (Apps Script)│
                                                          │  Kanban + טופס עדכון   │
                                                          │  דשבורד + תזכורות     │
                                                          └──────────────────────┘
```

1. **הקובץ החי** — נשאר כפי שהוא, מקור-אמת בתקופת המעבר. **לא כותבים אליו.**
2. **App DB** — Google Sheet **חדש ונפרד**, סכמה מנורמלת (טבלה אחת `פניות` + טאבי lookup). ה-backend של האפליקציה.
3. **Web App** — Apps Script Web App (כמו `admin.html` ב-supervisor-match) שקורא/כותב ל-App DB בלבד.

**ה-importer** הוא לב הבטיחות: סקריפט שקורא את הקובץ החי, **מונה את כל הלשוניות (כל לשונית = תוכנית)**, מפענח כל אחת לפי שורת-הכותרת שלה, ו-upsert לתוך App DB לפי `קוד לקוח`/שם+טלפון, עם `תוכנית` = שם הלשונית. רץ לפי דרישה — **קריאה בלבד מהקובץ החי**, אפס כתיבה אליו. תוכניות חדשות (לשוניות חדשות) נקלטות אוטומטית.

## 5. אסטרטגיית המעבר (3 שלבים) — מבטיחה אי-פגיעה + איחוד ברור

### Phase 1 — מקביל, קריאה + הרחבה (לא נוגעים בקובץ)
- בונים App DB + Web App.
- ה-importer מסנכרן קובץ-חי → App DB (חד-כיווני).
- המזכירה **ממשיכה למלא בקובץ החי**. האפליקציה נותנת לה **תצוגה** יפה: משפך, דשבורד, לידים תקועים, תזכורות.
- כתיבה באפליקציה בשלב זה = רק **שדות הרחבה שהקובץ לא מכיר** (snooze תזכורת, הערת פעולה-הבאה), ממופתחים לפי קוד לקוח — בלי התנגשות עם ה-import הבא.
- **בלי סיכון:** אם משהו לא עובד, פשוט ממשיכים בקובץ.

### Phase 2 — כתיבה מלאה ב-App DB (עדיין מקביל)
- האפליקציה הופכת ל-read-write מלא על App DB: כל עדכוני המשפך נעשים באפליקציה.
- ה-importer ממשיך לרוץ למקרה שעדיין נכנס משהו בקובץ הישן, עם זיהוי-קונפליקטים (אם שתי הגרסאות שונות — מסמנים לסקירה ידנית).
- בודקים שהאפליקציה מחזיקה הכול נכון מול הקובץ.

### Phase 3 — Cutover (איחוד סופי)
- מיגרציה אחרונה: קובץ-חי → App DB כ-master.
- מכבים את ה-importer. הקובץ הישן הופך לארכיון/גיבוי (read-only).
- מכאן והלאה **רק האפליקציה**. תיעוד ה-cutover נשמר כאן.

## 6. סכמת App DB — טבלת `פניות` מנורמלת

עמודה אחת לכל עובדה אטומית. אבני-דרך = ✓ + תאריך + מס' קבלה.

**א. זיהוי**
`id` · `תאריך פנייה` · `שם` · `תוכנית` (כוח הצמצום / תלת-שנתית) · `מחזור` (תשפ"ז / כא תשפ"ו...) · `ערוץ פנייה` (אתר/טלפון/וואטסאפ/המלצה) · `קוד לקוח (CRM)`

**ב. קשר**
`טלפון` · `דואל` · `מגורים` · `עיסוק`

**ג. סף קבלה** (checkbox)
`תואר שני` · `ניסיון שנתיים` · `עבר במכון`

**ד. תהליך** (✓ + תאריך)
`נשלח חומר` · `שוחחנו` · `טופס הרשמה` · `מסמכי תעודות` · `המלצות` · `טפסים חסרים` (טקסט) · `מצב ראיון` (לא נקבע/נקבע/בוצע/התקבל/נדחה) + `תאריך ראיון`

**ה. תשלומים** (לכל אחד: ✓ + סכום + תאריך + מס' קבלה)
`דמי ראיון` · `דמי הרשמה` (300) · `מקדמה` (1200) · `שכ"ל` (4900) · `טופס הנחה` + `אחוז הנחה`

**ו. תוצאה**
`מכתב תוצאה` (נשלח/לא) · `סטטוס` · `סיבה שאין עניין` · `נרשם בCRM` (✓)

**ז. מחושב (Apps Script — לא להקלדה)**
`שלב במשפך` · `ימים מאז עדכון` · `פעולה הבאה` · `תאריך עדכון` (אוטו-חותמת)

**טאבי lookup:** `סטטוסים` · `שלבי משפך` · `סוגי תשלום` · `תבניות הודעה` · `הרחבות אפליקציה` (snooze/הערות).

## 10. תוכנית ביצוע — מה נבנה ב-Phase 1

1. ליצור את **App DB** (Google Sheet חדש) עם הסכמה מסעיף 6 + טאבי lookup + data-validation + עיצוב מותנה.
2. לכתוב `importer.gs` — קורא את הקובץ החי, מפענח את 8 הטבלאות, ו-upsert ל-App DB.
3. לכתוב את ה-Web App (`Code.gs` + `index.html`): לוח Kanban של פניות לפי שלב, לחיצה→כרטיס עדכון, דשבורד (משפך לכל תוכנית + תשלומים פתוחים + לידים תקועים).
4. אוטומציות: אוטו-חותמת תאריך, חישוב שלב/פעולה-הבאה, דגל תזכורות, כפתור "שלח חומר" (Gmail).
5. בדיקה מול הקובץ החי (read-only) — לוודא שהמספרים תואמים.

**נסגר (2026-06-17):** תוכניות = כל לשונית · משתמשים = טל/מיכל/ניהול · Gmail = חשבון מכון רוטנברג.

**פתוח אחד לפני Phase 1 — גישה לחשבון רוטנברג:**
מאחר שהאפליקציה רצה תחת חשבון המכון, יש שתי דרכי בנייה:
- **(א) מקצה-לקצה** — נועם נותן לי גישה זמנית לחשבון רוטנברג, ואני מקים הכול (App DB + Apps Script + deploy) שם.
- **(ב) מסירה** — אני כותב את כל הקוד והסכמה כאן בריפו, ונועם/טל מדביקים ומפעילים תחת החשבון (כמו ש-supervisor-match הוקם). אני מספק הוראות התקנה צעד-אחר-צעד.
ברירת מחדל מומלצת: **(ב)** — בלי שיתוף סיסמאות, ותואם לאופן שבו supervisor-match כבר עובד.

- תבניות-הודעה ורשימות-קשר מעורבבות בתוך טבלאות הפניות.

### תיקונים שבוצעו (סבב 2)
1. **סינון זבל** — שורה = רשומה רק אם יש טלפון/מייל/קוד לקוח (מנקה שורות-תבנית).
2. **תשלומים** — `toBool_` מזהה ערך מספרי כ"שולם" + שמירת הסכום בשדה הסכום.
3. **חילוץ קוד CRM** שדחוס בעמודות אחרות (חיפוש `2XXXXX` / "קוד לקוח:").
4. **"סטטוס" — מופע אחרון** בלשונית התעודה (הסטטוס האמיתי, לא ההערה).
5. **מפתח זהות לפי תוכנית** — אדם בשתי תוכניות = שתי רשומות (החלטת נועם).
6. **`recordType`** (פנייה / איש-קשר) — לשונית 3 מיובאת כרשימת קשר נפרדת; המשפך והדשבורד מסננים אותה החוצה; לשונית "אנשי קשר" באפליקציה מציגה אותה.

### החלטות (נועם, 2026-06-17)
- **אנשי חינוך** = רשימה נפרדת (לא במשפך). ✅
- **אדם בשתי תוכניות** = שני כרטיסים. ✅
- **כיוון אסטרטגי** = בעתיד **cutover**: אחרי שטל (מזכירה) תעבור על הממשק ותעיר — האפליקציה תהפוך למקור-האמת לשינויים. **מעצבים מעכשיו כאילו האפליקציה היא המרכז** (טבלה מנורמלת אחת + דיסקרימינטור recordType + lookups).
